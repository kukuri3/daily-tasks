## 1. システム概要

### 1.1 目的
大学業務のOutlookメールから未完了タスクを自動抽出し、優先度付きのインタラクティブなHTMLタスクリストとしてGitHub Pagesに公開する。外出先からもスマートフォンで確認可能。

### 1.2 アーキテクチャ

```
[Outlook Web]
    ↓ Claude in Chrome (MCP)
    ↓ 受信トレイ + 送信トレイをスキャン
[Claude Sonnet 4]
    ↓ タスク抽出・スレッド完結判定・要約生成
[tasks_cache.json]
    ↓ キャッシュ（差分管理）
[generate_html.py + Jinja2テンプレート]
    ↓ HTML生成
[git push → GitHub Pages]
    ↓
[https://kukuri3.github.io/daily-tasks/]
```

### 1.3 主要コンポーネント

| コンポーネント | ファイル | 役割 |
|---|---|---|
| タスクキャッシュ | `tasks_cache.json` | 抽出済みタスクの永続化・差分管理 |
| HTML生成 | `generate_html.py` | キャッシュ→HTML変換、git push |
| HTMLテンプレート | `templates/index.html.j2` | Jinja2テンプレート（UI） |
| メニューバーアプリ | `menubar_app.py` | macOS常駐アプリ（rumps） |
| 定期実行 | `com.daily-tasks.generate.plist` | launchd（平日8:00/13:00） |
| スケジュールタスク | Claude Code scheduled-tasks | Outlookスキャン＋ビルド |

### 1.4 HTML機能一覧

- チェックボックス（localStorage + GitHub API同期）
- チェック時フェードアウトアニメーション、リロード時非表示
- 優先度別セクション（🔴緊急 / 🟡近日中 / 🔵予定・軽微）
- ▶ メール要約トグル（展開/折りたたみ）
- ✉ メール本文モーダル表示
- 🔄 手動スキャンボタン
- ⚙️ GitHub PAT設定（ブラウザ間のチェック状態同期）
- スレッドステータス表示（open / waiting / resolved）
- 完了タスクのサマリーセクション（ページ下部）
- スマートフォン対応レイアウト
- ヘッダーにスキャン日時・ビルド日時表示

### 1.5 スキャン手順（送信トレイ突合方式）

初期実装では受信トレイのみをスキャンしていたが、スレッド完結判定の精度が不十分だった。
改善として**送信トレイも同時にスキャン**し、以下のルールで完結判定を行う：

| 送信トレイ | 受信トレイ（相手の反応） | 判定 |
|---|---|---|
| 返信なし | — | `open`（未着手） |
| 返信あり | 相手の反応なし | `waiting`（返答待ち） |
| 返信あり | お礼/了解/完了の返信あり | `resolved`（完結） |

この方式により、3/22の再スキャンでは21件中6件を正確に`resolved`と判定できた。

## GPT-OSS 120B について

8GB RAMのマシンでは**実用不可**でした：
- 65GBのモデルを仮想メモリで読み込むため、約60GBのスワップが発生
- 30分以上経過しても意味のある出力が得られず中断
- 推定所要時間: 数時間〜半日

**推奨:** GPT-OSS 120Bを使うには最低48GB RAM、理想的には64GB RAMが必要。

## 最終判断：ローカルLLMの採用見送り

上記の比較評価の結果、**ローカルLLMは本システムの要件を満たさない**と判断し、採用を見送った。

### 見送り理由

1. **精度不足（Qwen 2.5 14B）**
   - スレッド完結判定の正答率が60%（5件中3件）で、実用に耐えない
   - 「返信済み＝完結」という短絡的な判断により、日程調整中のタスクを見落とすリスクがある
   - スレッド間の関連（例：共同研究と特許が同一企業）を全く検出できない
   - 要約が1行程度と浅く、業務判断に必要な情報（期限、関係者、経緯）が欠落

2. **性能不足（GPT-OSS 120B）**
   - 8GB RAMのMacBook Airでは動作不可（65GBモデルのスワップで30分以上経過しても出力なし）
   - 高精度なローカルモデルを動かすには最低48GB RAMが必要で、現有ハードウェアでは対応不可

3. **ハイブリッド運用の複雑さ**
   - 日次はQwen、週次はClaudeというハイブリッド運用も検討したが、2つのパイプラインの保守コストに見合わない
   - Qwenの誤判定を前提に運用すると、結局Claudeで再チェックが必要になり二度手間

### 採用方針

**Claude Sonnet 4（API）で全面運用する。**

- Maxプラン契約中のため実質追加コスト0円
- ステータス判定・要約・関連スレッド検出のいずれもClaudeが圧倒的に優位
- 将来的にローカルLLMの精度が向上し、かつRAM増設したマシンに移行した場合は再検討の余地あり

### 削除済みソフトウェア

評価完了後、以下をすべて削除した：
- Ollama v0.18.2（アプリ、データディレクトリ、キャッシュ、設定）
- Qwen 2.5 14B モデル（9.0GB）
- GPT-OSS 120B モデル（65GB）
- 合計約74GBのディスク領域を回収

## 3. 今後の課題・検討事項

1. **Microsoft Graph API への移行**: Azure ADアプリ登録（IT管理者承認待ち）が完了すれば、ブラウザ操作なしでメール取得が可能になり、安定性・速度が大幅に向上する
2. **メニューバーアプリからのスキャン**: 現状はClaude Codeセッション内でのみOutlookスキャンが可能（MCP制約）。Graph API移行で解消予定
3. **キャッシュの自動クリーンアップ**: 1ヶ月以上経過したresolvedタスクの自動アーカイブ
4. **複数デバイス対応**: GitHub APIによるチェック状態同期は実装済み。PATの安全な管理方法の検討
